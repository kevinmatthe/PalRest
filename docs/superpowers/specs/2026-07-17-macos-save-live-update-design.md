# macOS 保存后可见性与配置热更新设计

## 问题

设置窗口和悬浮窗口是两个独立 WebView。当前保存只更新设置窗口内的 React 状态，悬浮窗口不会收到新配置，因此继续轮询旧 UID/服务地址，直到 WebView 被手动重载。

保存锁定设置还会触发生命周期 `Lock`。macOS 进程检测只识别原生 `Palworld.app`，无法可靠覆盖 Whisky、CrossOver 或远端数据场景；检测为 false 时锁定会隐藏悬浮窗口，调整模式则因强制可见而重新显示。

## 已确认行为

- Windows 保留现有的本地游戏进程显隐策略。
- macOS 不使用本地 Palworld 进程作为隐藏门槛；存在有效配置时，锁定模式也保持悬浮窗口可见和鼠标穿透。
- 设置成功持久化后，原生层向悬浮窗口发送规范化后的完整配置。
- 悬浮窗口收到配置后解析并替换当前配置，使旧 poller 停止、新 poller 立即按新服务地址、游戏和 UID 启动。
- 配置事件无效时不替换当前有效配置，也不泄露错误详情。
- 调整模式、位置持久化、Windows 生命周期和首次设置流程保持不变。

## 数据流

1. 设置窗口调用 `save_config`。
2. 原生层原子保存可编辑字段，并重新读取包含最新原生几何信息的配置。
3. 原生层仅向 `overlay` WebView 发出 `overlay-config-changed`。
4. 悬浮窗口校验事件负载并更新 `LiveOverlay` 的配置；React 清理旧 `SnapshotPoller`，再创建新 poller。
5. 设置窗口随后同步锁定/调整状态；macOS 锁定保持窗口可见并启用鼠标穿透。

## 错误处理与测试

- 保存失败时不广播配置。
- 广播失败使保存命令返回失败，设置窗口显示现有的同步错误，不假装完全成功。
- 前端回归测试覆盖配置事件触发新 UID 轮询及旧请求终止。
- Rust 回归测试覆盖 macOS 锁定后保持可见、Windows 仍按进程显隐，以及保存事件所需命令契约。
- 最终运行前端全量测试/构建、Rust 测试/Clippy/格式检查和 Go 全量测试；原生 macOS 行为仍需 CI 构建后的实机冒烟测试。
